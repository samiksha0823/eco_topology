## 1. Purpose & Audience

This brief defines the design direction for the Eco-Topology alert dashboard — the surface where conservation NGOs, forest departments, and field rangers consume early-warning alerts generated by the TDA + ML + LLM pipeline. The dashboard is the product's primary point of human contact; the sophistication of the underlying persistent-homology math must translate into something a non-technical field user can act on in seconds.

| Audience | Technical fluency | Primary task in the UI |
|---|---|---|
| Field ranger | Low — no data-science background assumed | Read one alert, understand where and roughly why, act on it |
| NGO analyst | Medium — comfortable with maps and dashboards | Triage a ranked list, prioritize investigation across a region |
| Forest department officer | Medium — needs defensibility, not math | Pull evidence for enforcement decisions and reporting |
| Research collaborator | High — comfortable with model internals | Inspect raw topological features and model performance |

## 2. Design Principles

1. Plain language first. The LLM-generated brief is the default view; the underlying math is available but never required reading.
2. Map is the home screen. Geolocation is the primary organizing structure — every alert exists somewhere before it exists as data.
3. Show the evidence trail. Every alert must let a skeptical user drill from “what” to “why” to “raw data,” without forcing that drill-down on everyone.
4. Calibrated urgency, not alarm fatigue. Visual weight (color, ordering) should track risk score; low-confidence or borderline cells should look calm, not red.
5. Consistent with the project's existing visual identity. Reuse the pitch deck's palette and typographic voice so the dashboard reads as the same product, not a bolted-on tool.

## 3. Visual Identity

Carried over from the Eco-Topology pitch deck for continuity across pitch → product:

| Token | Value | Usage |
|---|---|---|
| Background (dark surfaces) | #120B26 / #1C1340 | App shell, map basemap chrome, primary dashboard background |
| Card surface | #241A4D | Alert cards, panels |
| Emerald (primary accent) | #34D399 | Healthy / low-risk indicators, primary actions, links |
| Teal (secondary accent) | #5EEAD4 | Data visualization accents, connective UI elements |
| Gold (attention accent) | #FBBF24 | High-risk alerts, the LLM-reporting layer, anything needing visual priority |
| Lavender (muted text) | #B8AEDC | Secondary/supporting text on dark surfaces |
| Headline font | Cambria | Titles, section headers, key numbers |
| Body font | Calibri | Body copy, descriptions, table content |

> **Why gold for risk, not red**
>
> Red is overused for “error” states across most software and reads as alarm rather than information. Reserving gold specifically for elevated ecological risk keeps the palette calm and lets true high-priority alerts stand out without the UI feeling like a warning siren by default.

## 4. Information Architecture

- Map / Alert Overview (home) — geolocated risk map + ranked alert list
- Alert Detail — single-cell deep dive: LLM brief, risk score, evidence, history
- Region Summary — aggregate view across a monitored area (counts, trend over time)
- Model & Data Panel (research-collaborator view) — feature importances, validation metrics, raw persistence diagram
- Settings — alert threshold, monitored regions, notification preferences

## 5. Key Screens

### 5.1 Map / Alert Overview (Home)

Primary screen for NGO analysts and forest department officers. A dark basemap fills the viewport with alert cells rendered as colored markers sized/colored by risk_score (emerald → low, gold → high). A collapsible side panel lists alerts ranked by risk score, mirroring the map.

- Layout: full-bleed map, right-docked ranked list panel (collapsible on mobile), top bar with region selector and alert-threshold indicator.
- Each list row: risk badge, cell centroid location label, one-line LLM summary, relative time since last GEDI acquisition.
- Clicking a map marker or list row opens the Alert Detail panel without a full navigation away from the map (preserve spatial context).
- Empty / low-alert state should read as reassuring, not broken — e.g. “No elevated-risk cells in this region as of [date].”

### 5.2 Alert Detail

The core trust-building screen. Structured to answer, in order: what is happening, how confident is the system, and what is the evidence.

1. Header: location (lat/lon + nearest named place if available), risk score as a plain badge (e.g. “High risk — 0.87”), timestamp of the GEDI acquisition used.
2. LLM brief: 2–4 sentences in plain language, e.g. “This area shows a persistent structural gap in the canopy consistent with selective logging, distinct from natural variation. Confidence is high based on 120 LiDAR shots collected in this cell.”
3. Evidence drawer (collapsed by default): risk score breakdown, top contributing topological features (feature_importance), point count (n_points) as a confidence signal.
4. Advanced / research view (collapsed, gated to research-collaborator role): the raw persistence diagram, rendered as a birth–death scatter plot with the H1 max-persistence point highlighted.
5. Action row: mark reviewed, export report (PDF), open in external map app for field navigation.

> **Explainability, not just explanation**
>
> The Evidence drawer must always be reachable from the LLM brief in one click. A user who doubts the narrative should never hit a dead end — they should be able to see the exact numbers the LLM was given.

### 5.3 Region Summary

Aggregate view for tracking a monitored area over time — answers “is this region getting better or worse,” not just “what is the latest alert.”

- Trend chart: count of elevated-risk cells per pipeline run over time.
- Simple KPI row: cells monitored, active alerts, validated precision/recall for this region (from the last Hansen-validated run).
- Table of all cells in the region with sortable risk score, last-checked date, and status (new / persisting / resolved).

### 5.4 Model & Data Panel (Research View)

- Feature importance chart (bar) reusing the same emerald/teal/gold palette.
- Confusion matrix and precision/recall/F1/ROC-AUC from the latest Hansen-validated run.
- Raw feature table export (CSV) for offline analysis, matching the pipeline's X/y/meta contract.
- This view is intentionally denser and more technical — it is the one screen where TDA/ML vocabulary is appropriate.

### 5.5 Settings

- Alert threshold slider (maps directly to the pipeline's threshold parameter), with a live preview of how many current cells would newly qualify.
- Monitored region management (add/remove bounding boxes or named areas).
- Notification preferences (email/SMS digest frequency for new high-risk alerts).

## 6. Interaction Patterns & Content Guidelines

### 6.1 LLM Brief Writing Rules

- No TDA/ML jargon in the default brief (no “persistence diagram,” “Vietoris–Rips,” “H1 homology”) — reserve that language for the Advanced view.
- Always state the confidence basis in plain terms (e.g. number of LiDAR shots), never just a bare probability.
- Never state a number the structured model output didn't provide — the brief restates, it doesn't estimate.
- Keep briefs to 2–4 sentences; longer explanation belongs in the Evidence drawer, not the headline text.

### 6.2 Risk Visualization

- Use a continuous emerald → gold scale for risk_score, not a hard red/green binary, so borderline cells read as genuinely borderline.
- Marker size can additionally encode n_points (confidence), so a small high-risk marker visually signals “watch this, but with lower confidence.”

### 6.3 Accessibility

- Do not rely on color alone to convey risk — pair every risk badge with a text label (“High / Medium / Low”).
- Maintain WCAG AA contrast for body text on the dark navy background (verify Lavender #B8AEDC body text meets contrast at chosen sizes; increase size or lighten if not).
- Map interactions must have non-hover / non-mouseover equivalents for keyboard and touch users (field use is often mobile/outdoor).
- Support offline-friendly PDF export of Alert Detail for field use in low-connectivity areas.

## 7. UX Success Metrics

- Time-to-comprehension: a first-time, non-technical user can correctly state why a cell was flagged within 15 seconds of opening Alert Detail.
- Drill-down usage: proportion of alert views where a user opens the Evidence drawer — tracks whether the LLM brief alone is trusted or whether users feel the need to verify.
- Task completion: analyst can go from dashboard load to identifying the top 3 priority cells in under 60 seconds.
- Field usability: ranger can export and reference an Alert Detail PDF without network access.
